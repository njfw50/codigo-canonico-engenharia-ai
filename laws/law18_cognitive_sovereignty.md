# Canon XVIII: The Doctrine of Cognitive Sovereignty
## The Eradication of Cognitive Debt and "Vibe Coding"

### Premise
While Technical Debt resides in the codebase, Cognitive Debt resides in the atrophy of the developer's mind. The unscrutinized acceptance of AI-generated code ("Vibe Coding") results in "Cognitive Surrender," wherein the human operator loses the "Theory of the System." This Law establishes strict protocols to preserve human cognitive dominance over the machine, utilizing the method of Liturgical Cognitive Annotation.

### Article 18.1 — The Prohibition of Vibe Coding
No block of code generated by an AI assistant or autonomous agent shall be merged into the active repository based solely on operational success (i.e., "it compiles and the tests pass"). Code merged without deep human comprehension of its underlying logic is classified as a Critical Cognitive Violation.

### Article 18.2 — The Liturgical Cognitive Annotation Protocol
To guarantee the retention of the system's mental model, the method of **Liturgical Cognitive Annotation** must be strictly applied. 
By default, **the AI agent is mandated to automatically generate these annotations** (line-by-line or logically grouped blocks) whenever it produces non-trivial logic, unless the human engineer explicitly demands to write the annotations themselves. 

These comments must not merely repeat the syntax (what the code does); they must act as a clear, semantic, and "liturgical" reading of the architecture, explaining:
1. The exact cognitive intent and reasoning behind the operation.
2. How the operation connects to the broader Business and Architectural context.

### Article 18.3 — Cognitive Offloading vs. Cognitive Surrender
The use of AI is restricted strictly to **Cognitive Offloading** (delegating repetitive typing, boilerplate, and syntax recall). **Cognitive Surrender** (outsourcing the architectural design, security reasoning, and structural problem-solving to the AI without a preceding human rationale) is strictly forbidden. 

### Article 18.4 — Reconstructive Code Review
Code reviews are no longer just syntactic checks; they are **Reconstructive Reviews**. The reviewing engineer (or the human verifying the AI) must be able to mentally and verbally reconstruct the execution path of the code, guided by the Liturgical Annotations. If the code is too obscure to be read cognitively, it must be deleted and rewritten.

### Architectural Impact
Transforms code from mere machine instructions into a continuous, human-readable cognitive document. It acts as an absolute firewall against the cognitive atrophy caused by LLMs, ensuring that the team never loses sovereignty over their own repository architecture.
